2. Alur Pipeline

Setiap email yang masuk diproses dalam 7 langkah:

  1. Ingest — Email masuk via Mailpit (SMTP), REST API, atau IMAP fetcher
  2. Enqueue — Push ke Redis list email_pipeline
  3. Dequeue — Worker BLPOP dari queue (timeout 5 detik)
  4. Parallel Scoring — SpamAssassin + ML Classifier berjalan bersamaan via asyncio.gather()
  5. Fusion — Decision engine menggabungkan 3 skor jadi satu fused_score
  6. Routing — Label CLEAN / WARN / QUARANTINE berdasarkan threshold
  7. Output — Simpan ke DB + broadcast via WebSocket + alert (jika QUARANTINE)

3. Tiga Lapisan Deteksi

Layer 1 Supervised — XGBoost + TF-IDF

  • Model: XGBoostClassifier, 500 trees, GPU CUDA 12.9
  • Fitur: 28 structured features + 50,000 TF-IDF vocabulary
  • Akurasi: 100% pada testing dataset 105k email
  • False Positive Rate: 0.19%
  • Latency: ~16ms/prediction (warm)

Layer 2 Unsupervised — Isolation Forest + One-Class SVM

  • Dilatih HANYA dengan email bersih (49,048 samples)
  • Mendeteksi anomaly / zero-day / unknown threats
  • Fitur: 20 structured features (tanpa TF-IDF)
  • Fusion: IForest score × 0.6 + OCSVM score × 0.4

Layer 3 Rule-based — SpamAssassin

  • Skor raw -2.0 sampai 20+ berdasarkan ribuan rules
  • URIBL, DKIM, SPF, Bayesian filter, dll
  • Dinormalisasi ke [0, 1] untuk fusion

  • request_for_transfer — Deteksi permintaan transfer dana / wire
  • sender_reputation — 1.0 jika domain != lodaya.id, 0.0 jika domain trusted

5. Decision Engine (3-Way Fusion)

Input:
  sa_score       — SpamAssassin raw score (-2 to 20+)
  ml_probability — XGBoost probability (0.0 to 1.0)
  anomaly_score  — IForest + OCSVM (0.0 to 1.0)
  spf/dkim/dmarc — boolean hasil autentikasi

Step 1: Normalisasi SA
  sa_norm = clamp(sa_score, 0, 20) / 20

Step 2: Hard Override
  IF sa >= 15 OR ml >= 0.95 OR anomaly >= 0.90
     → QUARANTINE (fused = 1.0)

Step 3: Weighted Fusion
  fused = ml × 0.50 + sa_norm × 0.25 + anomaly × 0.25

Step 4: Auth Bonus
  IF spf & dkim & dmarc pass AND ml < 0.5
     → fused = max(0, fused - 0.10)

Step 5: Label
  fused < 0.30 → CLEAN
  0.30 - 0.70 → WARN
  ≥ 0.70      → QUARANTINE

10. Production Architecture (MX Record)

Untuk produksi nyata, system harus berada di jalur antara pengirim dan inbox — bukan berjalan paralel.

INTERNET │ DNS MX: mail.domain.com │ ┌────────▼────────┐ │ SMTP Receiver │ port 25 │ (smtp_receiver) │ └────────┬────────┘ │ Push ke Redis ┌────────▼────────┐ │ Pipeline Scan │ Layer 1+2+3 └────────┬────────┘ │ ┌──────────┼──────────┐ ▼ ▼ ▼ CLEAN WARN QUARANTINE │ │ │ ▼ ▼ ▼ Forward ke Forward + Tahan di Gmail/Outlook X-Header Dashboard

Cara Kerja Produksi

  1. MX Record — DNS domain Anda diarahkan ke server kita. Semua email dunia masuk ke SMTP Receiver kita duluan.
  2. SMTP Receiver — Menerima email asli via SMTP port 25, push ke Redis queue. File: worker/smtp_receiver.py
  3. Pipeline Scan — Worker proses: SA + ML + Anomaly → fusion → label
  4. Forward — Jika CLEAN/WARN, email diteruskan ke Gmail/Outlook asli via SMTP dengan TLS. File: worker/email_forwarder.py
  5. Karantina — Jika QUARANTINE, email tidak diteruskan, hanya muncul di dashboard.

Yang Harus Di-setup untuk Produksi

KomponenKeterangan

SMTP ReceiverJalankan python -m worker.smtp_receiver di port 25 (root)

Gmail App PasswordBuat App Password di Google Account → Security

11. Developer Guide

Menambahkan Fitur Baru

Untuk menambah fitur deteksi baru:

  1. Tambahkan field baru di classifier/features.py → class EmailFeatures
  2. Implementasi logika ekstraksi di FeatureExtractor.extract()
  3. Tambahkan nama fitur ke STRUCTURED_FEATURES list
  4. Generate ulang training data: python scripts/extract_100k.py
  5. Retrain model: python scripts/train_model.py
  6. Restart classifier API
